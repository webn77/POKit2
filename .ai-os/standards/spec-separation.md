# Spec Separation Standard

언제 별도 spec 이슈를 선행하고, 언제 카드 AC로 충분한지 판정하는 단일 기준.

PO 결정: 2026-06-12 "모든 이슈에 spec을 따로 잡을 필요는 없다. 지금 방식 유지 + 분리 기준만 명문화."

## 판정 기준 3종

| 기준 | 무슨 뜻 | 예시 |
|---|---|---|
| **1. 되돌리기 비싼 결정인가** | 한번 실행하면 되돌리기 어려운 구조 변경 또는 외부 배포 형태 변경 | 설치 토폴로지 변경, 공개 API 교체, npm 패키지 이름 변경 |
| **2. 여러 이슈가 그 결정에 의존하는가** | 이 결정이 없으면 후속 이슈의 AC나 범위를 확정할 수 없는 경우 | 구현 이슈 2건 이상이 동일 설계 문서를 consumes로 참조 |
| **3. 외부가 보는 계약인가** | npm 표면, 공개 README, 외부 제품 사용자가 보는 스펙 변경 | npm 패키지 설치 경로 변경, 공개 CLI 명령 인터페이스 개편 |

**판정 규칙: 셋 중 하나라도 해당하면 spec 이슈를 선행한다. 아무것도 해당하지 않으면 카드 AC로 충분하다.**

## 판정 결과 두 갈래

### spec 이슈 선행

- 별도 `issue_type: spec` 카드를 생성하고 실행 순서를 선행(`recommended_order` 낮게)으로 설정한다.
- spec 카드의 산출물(결정문 또는 설계 문서)이 후속 구현 카드의 `consumes`에 등재된다.
- spec 게이트가 통과된 뒤에만 후속 구현 이슈를 실행한다.

### 카드 AC로 충분

- 별도 spec 카드를 만들지 않는다.
- 이슈 카드 안의 AC와 Development Plan으로 설계 의도를 충분히 표현한다.
- 불필요한 spec 카드는 오버헤드(병목, 라우팅 복잡도, 두 카드 간 sync 부담)만 늘린다.

## 선례

### POK-338 — 작동 사례 (3기준 모두 해당)

`projects/pokit/issues/POK-338.md`

v0.19에서 "설치 토폴로지 스펙" 이슈를 별도로 선행한 사례. 3기준이 전부 해당했다:

- **되돌리기 비싼 결정**: 글로벌 단일 설치 vs 프로젝트 내 복사 설치를 한번 결정하면 기존 사용자 마이그레이션 경로가 달라진다.
- **다중 의존**: POK-340(구현)이 POK-338을 `consumes`로 직접 참조하고, POK-341(표면 정리)은 POK-340을 통해 그 토폴로지 결정에 의존했다. spec 없이는 두 이슈의 AC를 확정할 수 없었다.
- **외부 계약**: npm 패키지 설치 경로(`npx pokit2 install`)가 사용자가 보는 계약이었다.

결과: topology-spec.md 박제 → PO 승인 → 구현·표면 이슈가 결정 기준으로 착수. 한 스프린트 안에서 닫힘.

### v0.19 retro — 반증 사례 (spec 부재가 원인이 아닌 경우)

`docs/v0.19.0/retro.md`

v0.19 작업 중 드러난 결함(0.18.0이 품고 있던 fresh 설치 `issue:create` 차단·`doctor` fail 11건)을 retro는 "게이트 전 실측이 아니라 게이트 후(다음 이슈) 실측에서야 발견 — 실측 시점이 늦었다"로 분석했다. 원인은 "spec을 더 잡았어야 했다"가 아니다.

retro 회고 패턴 한 줄(line 33): "게이트 전 '진짜 사용자 경로 실측'(빈 폴더 설치 → 첫 명령)이 설계 리뷰가 못 잡는 결함·누출·사고를 전부 끌어냈다 — 실측은 게이트 조건이지 사후 확인이 아니다."

시사점: spec을 별도로 선행해도, 실측을 게이트 조건으로 두지 않으면 결함은 남는다. spec 이슈는 "결정의 불가역성"을 다루는 도구이지, 검증 지연의 대체재가 아니다.

## POK-147 흡수

`projects/pokit/issues/POK-147.md` — "IDD Methodology Standard: Sprint-SDD + Issue-IDD Hybrid"는 v0.10 scope 대기 중 park 상태로 남아 있다.

POK-147의 핵심 문제의식은 "Sprint-SDD(전체 설계 문서)와 Issue-IDD(개별 이슈 설계)의 경계를 어떻게 나누나"였다. 본 표준의 판정 기준 3종이 그 경계 질문에 실용 답을 제공한다. 별도 방법론 문서 없이, "기준 3종 중 하나라도 해당하면 spec 선행"이 이 경계 결정 규칙이다.

POK-147은 본 표준에 흡수 처리한다. POK-147 카드 자체의 frontmatter 갱신(absorbed 기록)은 main 세션이 별도 처리한다.

## Out of Scope

이 문서는 판정 기준만 박제한다. spec 이슈 내부의 작성 방식, 방법론 전체 재설계, IDD vs SDD 개념 정비는 다루지 않는다.
