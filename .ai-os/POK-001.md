## Acceptance Criteria

- Project key 또는 namespace가 명확하다.
- durable work는 Harness Issue에 연결된다.
- issue/task는 단일 파일로 시작한다.
- subtask는 MVP에서 `subtask_id`가 있는 subagent 결과로 기록한다.
- 임시 Todo는 실행 체크리스트이며 durable issue를 대체하지 않는다.
- Epic은 MVP의 1급 artifact가 아니다.
- issue-per-durable-change, small scoped changes, tests before gate claim 규칙을 따른다.
- no unrelated refactor, public-safe starter content, review evidence before completion 규칙을 따른다.

### Code Management Notes

- issue-per-durable-change: durable code, contract, docs, or release work must be tied to one Harness Issue.
- small scoped changes: keep each change reviewable in one gate.
- tests before gate claim: run fresh verification before claiming a behavior, contract, or release state.
- no unrelated refactor: do not mix cleanup or restructuring into an unrelated issue.
- public-safe starter content: starter files must avoid secrets, personal paths, private company assumptions, production history, run logs, and event receipts.
- review evidence before completion: completion needs local evidence; delegated agent output is only input evidence.

### PO Work Model Audit

- Project key support: covered by `namespace`.
- Epic support: MVP gap.
- Issue/task support: covered by single-file Harness Issue.
- Subtask/subissue support: MVP supports `subtask_id`; folder-based `subissue.yaml` is future scope.
- Temporary Todo boundary: Todo is execution support, not durable source of truth.
